🛡️ HCS-U7 Widget v3 — Test UX Scénarios

Simulation visuelle des 5 niveaux de décision du widget

⚠️ Rappel Architecture Le widget ne fait PAS de tests cognitifs (Stroop, N-Back…). Il collecte passivement des signaux comportementaux (souris, clavier, scroll, touch, gyroscope) et calcule un score de risque. Les actions ci-dessous sont déclenchées selon les seuils : allow (<30) → soft (30-55) → challenge (55-80) → hard_challenge (80-95) → bunker/block (≥95).
ALLOW

Utilisateur normal

Score : 0 – 29 / 100
Ce qui se passe : Absolument rien. Aucune UI visible, aucune friction. L'utilisateur ne sait même pas que le widget existe. La page se charge normalement.

Code exécuté : executeAllow() — un simple log debug invisible.
SOFT

Légèrement suspect

Score : 30 – 54 / 100
Ce qui se passe : Actions invisibles en arrière-plan. L'utilisateur ne voit rien.

Actions silencieuses :
• PoW-lite : burn CPU 150-300ms (coûteux à l'échelle bot)
• JS Attestation : vérification canvas + APIs navigateur
• Silent retry + token refresh si configuré
CHALLENGE

Bot suspecté (variabilité TR <30ms)

Score : 55 – 79 / 100
Ce qui se passe : Overlay modal avec slider. L'utilisateur doit déplacer le curseur sur la valeur cible (rendue via canvas, illisible par les bots DOM).

Anti-bot : min 800ms, min 3 input events, tracking micro-mouvements.
Échec → BLOCK immédiat.
BUNKER

Risque très élevé / Incident

Score : ≥ 95 / 100 (bunker activé)
Ce qui se passe : Plein écran sombre. Slider avec tolérance réduite (±3 vs ±5). Le site est complètement bloqué derrière la gate.

Réussite → Whitelist 15min (sessionStorage).
Échec → Retry illimité (pas de block).
BLOCK

Bot confirmé / Challenge échoué

Score : ≥ 95 (bunker désactivé) ou challenge échoué
Ce qui se passe : La page entière est remplacée par un écran de blocage plein écran. document.body est vidé et remplacé.

Message affiché : "Accès bloqué" + raison + branding HCS-U7.
Aucun redirect. Aucun lien de retour. L'utilisateur doit fermer l'onglet.
